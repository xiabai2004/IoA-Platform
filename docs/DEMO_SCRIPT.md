## 详细分镜

### 一、开场（~1分钟）

**画面**：标题卡 "IoA 分布式网络运维协同平台 — 基于智能体互联网的跨域协同调度系统"

**旁白**：
> 传统网络运维面临三个核心挑战：第一，跨地域设备分散，告警信息割裂，需要人工逐域排查；第二，故障定位依赖专家经验，平均MTTR达到数小时；第三，修复操作依赖手动执行脚本，缺乏闭环验证。
>
> 我们的系统——IoA分布式网络运维协同平台，通过智能体互联网架构，让多个AI Agent自动协同完成网络监控、故障诊断、自动修复和闭环验证，将故障响应时间从小时级压缩到秒级。

---

### 二、系统架构（~3分钟）

**画面**：分层架构图（自下而上展示）

#### 2.1 IoA中间件层（~1.5分钟）

**旁白**：
> 系统的核心是我们自研的IoA中间件，包含四个关键模块：
>
> **Agent注册中心**：所有Agent启动时自动注册能力描述，包含支持的协议、技能标签和负载状态。这是分布式的"服务发现"能力。
>
> **语义消息路由总线**：Agent之间不直接寻址，而是通过意图+能力匹配实现消息投递。比如orchestrator发送"需要网络监控"的消息，路由总线根据"network_monitoring"标签自动投递给monitor-global。
>
> **DAG任务调度引擎**：采用Kahn拓扑排序算法解析任务图，支持节点级重试、超时控制和全链路审计日志。
>
> **闭环验证器**：每个修复操作完成后，自动执行三态判定——pass（验证通过）、retry（局部重试）、fail（标记失败，人工介入）。

**画面切换**：代码片段展示 register、route、verify 三个核心接口

#### 2.2 Agent运行时层（~1.5分钟）

**旁白**：
> 上层是5个基于LangGraph构建的专业Agent：
>
> **orchestrator**接收自然语言运维指令，匹配DAG模板，将任务分发给各个Agent。比如用户输入"华东地区网络延迟异常"——orchestrator识别到关键词"延迟异常"和"华东"，自动匹配full_remediation模板。
>
> **monitor-global**负责全网四域实时指标采集，包括延迟、丢包率、带宽利用率、吞吐量和连接数。
>
> **diagnoser**收到异常指标后，调用DeepSeek大模型进行根因分析，区分链路拥塞、DDoS攻击、设备故障等故障类型。
>
> **repairer**执行修复操作——对于链路拥塞，触发流量工程重路由；对于DDoS攻击，启用流量清洗策略。
>
> **verifier**执行闭环验证，采集修复后指标与基准值对比。一个关键设计是"实时指标兜底"——如果故障已被清除、当前指标正常，即使前后对比improvement不满足阈值，也直接判定pass，避免死循环。

### 三、核心创新点（~2分钟）

**画面**：三个创新点依次以动画/卡片展示

#### 3.1 语义消息路由

> 传统Agent系统采用硬编码的点对点通信——Agent A直接调用Agent B的API。当系统规模增长到几十个Agent时，这种耦合会成为灾难。
>
> 我们的路由总线让Agent之间完全解耦。发送方只需描述"我需要什么能力"，路由总线根据注册中心的能力描述自动匹配目标Agent。这是真正的"智能体互联网"——Agent不需要知道彼此的存在。

**画面展示**：一对多路由示意图 + IoAP消息格式JSON

#### 3.2 DAG编排引擎

> 运维流程不是简单的线性管道。同一个DAG可能因为节点失败需要触发分支重试，不同故障类型可能走不同的子图。
>
> 我们的DAG引擎支持有向无环图的完整表示，每个节点可以声明依赖的前驱节点和后继节点，调度器自动计算拓扑序并执行。如果diagnoser返回retry建议，调度器自动回退到repair节点重新执行。

#### 3.3 闭环验证

> 修复≠恢复。很多自动化运维工具的终极问题是执行了修复操作，但没有验证修复是否生效。我们的verifier在每个DAG末尾强制执行验证——对比修复前后指标，确保服务质量恢复到正常范围。
>
> 特别地，verifier的实时指标兜底逻辑解决了时序问题：如果repairer在retry轮次发现故障已被清除，采集的before/after都是正常值，improvement接近0——此时不按阈值判定，而是检查当前指标是否达标，避免死循环。

### 四、演示1：链路拥塞端到端（~3分钟）

**画面**：GUI录屏，展示完整操作流程

#### 操作序列

1. 打开 https://xiabai.site/gui → 展示完整界面
2. 面板逐一介绍：拓扑图（四域正常状态）、指标仪表盘（华东延迟14ms）、Agent集群（5个在线）、消息流
3. 点击下拉框 → 选择"链路拥塞" → 选择"华东" → 点击**⚡ 注入故障**
4. **实时效果**：拓扑图中华东节点变红，指标面板华东延迟飙升到 179ms，丢包 5%，带宽 90%
5. 点击**▶ 执行指令**（预设指令："华东地区网络延迟异常，请全流程诊断修复"）
6. **DAG执行**：底部DAG面板自动展开，4个节点依次执行

#### 旁白（配合操作）

> 现在华东地区一切正常，延迟14毫秒，丢包0.2%。
>
> 我注入一个链路拥塞故障——可以看到，华东节点立刻变红，延迟飙升到179毫秒，丢包率跳到5%，带宽利用率接近90%。这就是一个典型的链路拥塞场景。
>
> 接下来，我发送一条完全口语化的指令："华东地区网络延迟异常，请全流程诊断修复"——注意，这不是API调用，就是一句人话。
>
> 系统自动创建了DAG，监控节点采集异常指标，诊断节点调用LLM判断为链路拥塞，修复节点执行流量调度，验证节点采集修复后指标——延迟从179毫秒降到10.8毫秒，丢包从5%降到0.12%，验证通过。整个过程不到6秒。

### 五、演示2：DDoS攻击 + 验证兜底（~2分钟）

**画面**：GUI录屏，展示DDoS场景

#### 操作序列

1. 清除故障 → 重新注入"DDoS攻击" → 华北
2. 展示华北指标异常（流量暴涨）
3. 发送指令 → DAG执行
4. 重点展示 verifier 的 verdict = pass

#### 旁白

> 第二个场景——DDoS攻击。注入到华北地区，可以看到流量激增。
>
> 这里我想特别展示我们的闭环验证器。在开发初期，DDoS场景遇到过死循环bug——repairer清除故障后，模拟器指标仍在高位波动，verifier采集到的before/after差异太小，不满足30% improvement阈值，导致无限retry。
>
> 我们增加了一个关键逻辑：实时指标兜底。如果当前指标已经正常——延迟低于100ms、丢包低于1%、带宽低于85%——就直接pass，不再校验improvement。这个fix让所有故障场景都能稳定收敛。
>
> 验证通过，华北恢复，DAG完成。

**旁白**：

>
> 长期愿景：将IoA中间件开放为通用的多Agent协同框架，任何需要AI Agent协同工作的场景——智能客服调度、供应链协同、智慧城市事件响应——都可以基于同一套语义路由和DAG引擎构建。

## 制作建议

### 录制工具
- 使用 OBS Studio 录制屏幕
- 分辨率：1920×1080（16:9）
- 帧率：30fps

### 后期要求
- 旁白录音清晰，语速适中（约 150字/分钟）
- 架构图以静态图片叠加到视频中，配合缩放和标注动画
- 代码片段展示控制在 10 秒以内，不要过长
- GUI 录屏保证鼠标操作清晰可见
- 加上时间戳/进度条方便评委跳转

### 注意事项
- 演示前确保服务正常运行（`systemctl status ioa-platform ioa-simulator`）
- 清除所有历史DAG记录，保持界面整洁
- 关闭浏览器其他标签页和个人书签栏
- 建议用 Chrome 无痕模式录制
- 禁止在视频中出现任何个人信息（微信、邮箱等）
